



# 1. The Core Problem DevOps Tools Solve

Before tools, understand the pain:

## Problem A: Infrastructure Chaos

Old way:

* Manually create servers
* Click in cloud console
* No record of what was done

Issues:

* Not reproducible
* Error-prone
* No version control

 Solution → **Infrastructure as Code (IaC)**
→ handled by Terraform

---

## Problem B: Configuration Drift

Even if servers exist:

* Different machines have different configs
* “Works on my machine” problem

 Solution → **Configuration Management**
→ handled by Ansible

---

## Problem C: No Visibility (Logs & Debugging)

When app fails:

* Where is error?
* Which server?
* What happened?

 Solution → Centralized logging
→ handled by ELK stack:

* Elasticsearch
* Logstash
* Kibana

---

# 2. Terraform – What Problem It Solves

### Core Idea

Define infrastructure in code:

```hcl
resource "aws_instance" "web" {
  instance_type = "t2.micro"
}
```

Instead of:

* Clicking AWS UI
* Manually configuring

---

## What Terraform Actually Does

### 1. Desired State System

You say:

```id="6mk2qq"
"I want 1 EC2 instance"
```

Terraform:

* Checks current state
* Creates/updates to match desired state

---

### 2. State Management

Keeps a file:

```id="62vl0i"
terraform.tfstate
```

Tracks:

* What exists
* What Terraform created

---

### 3. Execution Plan

```bash
terraform plan
```

Shows:

* What will change BEFORE applying

---

## Real Problem It Solves

* Multi-cloud provisioning
* Reproducibility
* Version-controlled infra
* Automation at scale

---

# 3. ELK Stack – What Problem It Solves

You already understand metrics. Logs are different.

---

## Why Logs Are Hard

Without ELK:

* Logs scattered across machines
* SSH into each server to debug
* No search capability

---

## ELK Solution

### Elasticsearch

* Stores logs
* Indexes them (like Google search)

 Fast querying:

```id="yqz0z0"
error AND payment_failed
```

---

### Logstash

* Collects + transforms logs

Example:

* Parse JSON logs
* Extract fields
* Add metadata

---

### Kibana

* UI for logs
* Dashboards
* Search interface

---

## Real Problem It Solves

* Debugging distributed systems
* Root cause analysis
* Audit logs
* Security monitoring

---

# 4. Terraform vs Ansible (CRITICAL CONCEPT)

Using:

* Terraform
* Ansible

---

## 4.1 Core Difference

| Aspect   | Terraform             | Ansible              |
| -------- | --------------------- | -------------------- |
| Purpose  | Create infrastructure | Configure systems    |
| Level    | Infra level           | OS/App level         |
| State    | Maintains state file  | No state (stateless) |
| Approach | Declarative           | Procedural (mostly)  |

---

## 4.2 Example (Important)

### Terraform:

```id="cfqw7d"
Create EC2 instance
```

### Ansible:

```id="nsgsvi"
Install nginx on that EC2
```

 That’s the separation

---

## 4.3 Can Ansible Replace Terraform?

### Short Answer:

**No (in real-world DevOps)**

---

### Why Not?

### 1. State Management

* Terraform tracks infra
* Ansible does not track infra lifecycle

 Ansible doesn’t know:

* What it created earlier
* What needs deletion

---

### 2. Infrastructure Complexity

Terraform:

* Handles dependencies
* Graph-based execution
* Cloud-native APIs

Ansible:

* Executes tasks sequentially

---

### 3. Idempotency vs Lifecycle

* Terraform → full lifecycle (create/update/destroy)
* Ansible → configuration enforcement

---

## 4.4 Where People Get Confused

Yes, Ansible *can*:

* Create EC2 instances

BUT:

* Not efficient for large infra
* No proper state tracking
* Hard to manage changes

 So:

* Possible ≠ Recommended

---

# 5. Real-World Workflow (How They Work Together)

```id="z0rzha"
Terraform → creates servers (EC2, VPC, DB)

Ansible → configures them (install nginx, docker, app)

App → generates logs

Filebeat → collects logs

Logstash → processes logs

Elasticsearch → stores logs

Kibana → visualizes logs
```

---

# 1. What You Actually Need for Terraform on AWS

Using Amazon Web Services

## Minimum Requirements

### 1. AWS Account

* Create account (one-time)

---

### 2. IAM User (IMPORTANT)

Using AWS IAM

Create:

* A **non-root IAM user**
* With programmatic access

---

### 3. Permissions

For learning:

* Attach policy:

```id="z4a1kn"
AdministratorAccess
```

For real-world (later):

* Use least privilege (custom policies)

---

### 4. Access Keys

You’ll get:

```id="sn9h2r"
Access Key ID
Secret Access Key
```

Used by:

```bash id="i9m4o3"
aws configure
```

---

# 2. How Terraform Uses This

Terraform does NOT log in like a user.

Instead:

* Uses AWS APIs
* Authenticates via credentials

Flow:

```id="0qk6g6"
Terraform → AWS API → Create resources (EC2, VPC, S3)
```

---

# 3. Do You Need to Create IAM via Terraform?

### For beginners:

 NO — create IAM manually first

---

### For advanced usage:

 YES — Terraform can manage IAM too

Example:

```hcl id="x5e1gk"
resource "aws_iam_user" "devops_user" {
  name = "devops-user"
}
```

---

# 4. Recommended Setup Strategy

## Step 1 (Manual – one time)

* Create AWS account
* Create IAM user
* Generate keys

---

## Step 2 (Local machine)

```bash id="7j9vsm"

```

---

## Step 3 (Terraform takes over)

* Create EC2
* Create VPC
* Create S3
* Create Security Groups
