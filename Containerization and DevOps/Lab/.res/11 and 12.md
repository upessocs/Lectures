# **PART A – CONCEPT CONTINUATION**

### From Experiment 6, students already know:

* `docker run` → single container (imperative)
* Docker Compose → multi-container (declarative)
* WordPress + MySQL setup



## **New Concept Introduced**

> Compose is NOT real orchestration
> Swarm adds orchestration capabilities



## **What is Orchestration?**

Orchestration means:

* Managing multiple containers automatically
* Scaling services
* Self-healing (restart failed containers)
* Load balancing



## **Transition Flow**

```text
Docker Run → Compose → Docker Swarm (Stack)
```

# **PART B – PRACTICAL (EXTENSION OF EXPERIMENT 6)**

## **Task 1: Reuse Existing Compose File**

Use the same `docker-compose.yml` from Experiment 6.


## **Task 2: Initialize Swarm**

```bash
docker swarm init
```

Verify:

```bash
docker node ls
```



## **Task 3: Deploy as Stack**

```bash

```



## **Task 4: Verify Services**

```bash

```

```bash

```



## **Key Observation**

* Containers are now **services**
* Managed by Swarm, not manually



## **Task 5: Scale Application**

```bash
docker service scale wpstack_wordpress=3
```



## **Check scaling**

```bash
docker service ls
```



## **What to observe**

* Multiple replicas running
* No port conflict
* Internal load balancing works



## **Task 6: Test Self-Healing**

1. Find a container:

```bash
docker ps
```

2. Kill it:

```bash
docker kill <container_id>
```

3. Observe:

```bash
docker service ps wpstack_wordpress
```

# **PART B – PRACTICAL (Using k3d / Minikube)**

Students already know:

* `kubectl`
* k3d cluster
* Minikube

So we go straight to deployment.

---

## **Task 1: Create Deployment**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: wordpress
spec:
  replicas: 2
  selector:
    matchLabels:
      app: wordpress
  template:
    metadata:
      labels:
        app: wordpress
    spec:
      containers:
      - name: wordpress
        image: wordpress:latest
        ports:
        - containerPort: 80
```

Apply:

```bash
kubectl apply -f wordpress-deployment.yaml
```

---

## **Task 2: Expose Service**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: wordpress-service
spec:
  type: NodePort
  selector:
    app: wordpress
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30007
```

Apply:

```bash
kubectl apply -f wordpress-service.yaml
```

---

## **Task 3: Verify**

```bash
kubectl get pods
kubectl get svc
```

Access:

```
http://<node-ip>:30007
```

---

## **Task 4: Scaling**

```bash
kubectl scale deployment wordpress --replicas=4
```

---

## **Task 5: Self-Healing**

```bash
kubectl delete pod <pod-name>
```

✔ Pod recreated automatically

# **PART D – ADVANCED LAB (IMPORTANT)**

## Your Question:

> Can we setup real cluster using kubeadm?

✔ **Yes — and it’s a GREAT demo**

---

## **Kubeadm-based cluster (real Kubernetes)**

Using:

* kubeadm
* kubelet

---

## **Minimal Lab Setup**

### Option 1 (Best for lab):

* 2–3 VMs (VirtualBox / VMware)
* Or multiple nodes in LAN

---

### Steps (High Level)

### 1. Install components

```bash
apt install kubeadm kubelet kubectl
```

---

### 2. Initialize master

```bash
kubeadm init
```

---

### 3. Setup kubeconfig

```bash
mkdir -p $HOME/.kube
cp /etc/kubernetes/admin.conf $HOME/.kube/config
```

---

### 4. Install network plugin (important)

Example:

```bash
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
```

---

### 5. Join worker nodes

Command provided by `kubeadm init`:

```bash
kubeadm join <master-ip>:6443 --token ...
```

---

### 6. Verify cluster

```bash
kubectl get nodes
```

---

## **Important Teaching Insight**

| Tool     | Purpose                       |
| -------- | ----------------------------- |
| k3d      | Lightweight learning          |
| Minikube | Single-node cluster           |
| kubeadm  | Real production-style cluster |
